# バックアップシステム構築の記録

coconikkiに自動バックアップシステムを導入した際の開発記録です。

---

## 🎯 きっかけ

サンプル交換日記機能の実装とデプロイが完了し、coconikkiが本格的に動き始めた段階で、ふと気づきました。

**「もしデータが消えたらどうしよう？」**

VPSで運用している以上、ハードウェア障害、誤操作、その他予期せぬトラブルでデータが失われるリスクは常にあります。ユーザーの大切な投稿や画像を守るため、バックアップシステムの構築が急務でした。

---

## 📋 要件定義

### 絶対条件

1. **追加コスト0円**
   - 個人プロジェクトのため、バックアップに追加費用をかけられない
   - 既存の無料リソースを活用したい

2. **完全自動化**
   - 手動バックアップは忘れる可能性が高い
   - cron等で完全自動化したい

3. **復元可能性**
   - バックアップを取るだけでなく、実際に復元できることが重要
   - 緊急時にすぐ使える復元手順が必要

### バックアップ対象

- PostgreSQLデータベース（全テーブル）
- Active Storage画像（ユーザーアップロード画像）

### あったら嬉しい機能

- バックアップ成功/失敗の通知
- 古いバックアップの自動削除
- 複数バックアップ先への対応（将来的に）

## 🤔 技術選定

### バックアップ先の検討

候補として以下を検討しました：

#### 1. VPS内ローカルストレージ
- **メリット**: 追加設定不要、高速
- **デメリット**: VPS故障時に一緒に失われる ❌

#### 2. AWS S3
- **メリット**: 信頼性が高い、将来的なスケーラビリティ
- **デメリット**: 無料枠はあるが、長期的にコストがかかる可能性 ⚠️

#### 3. Google Drive
- **メリット**:
  - 15GB無料枠（個人的にGoogle One契約で70GB利用可能）
  - 追加コスト0円
  - rcloneで簡単にアクセス可能
  - Webブラウザで手動確認も可能
- **デメリット**: APIレート制限がある（が、1日1回のバックアップでは問題なし）

**→ Google Driveを採用！**

### バックアップツールの選定

#### rclone
- **選定理由**:
  - Google Drive含む70以上のクラウドストレージに対応
  - OAuth認証をサポート
  - コマンドラインで完結
  - 古いファイルの自動削除機能（`--min-age`）

#### PostgreSQL標準ツール（pg_dump）
- 理由: PostgreSQL公式の信頼性の高いツール

## 🛠️ 実装プロセス

### Phase 1: 基本バックアップスクリプト作成

最初に、シンプルなバックアップスクリプトを作成しました。

**やったこと**:
- PostgreSQLダンプ（gzip圧縮）
- Active Storageのtar.gz作成
- Google Driveへのアップロード（rclone）
- 古いバックアップの自動削除（7日分保持）

**学び**:
- `pg_dump`は簡単だが、権限設定に注意が必要
- Active StorageはDockerボリューム内にあるため、sudoが必要
- rcloneの`--min-age`オプションで古いファイルを簡単に削除できる

**ハマったポイント**:
- Active Storageのパス: `/var/lib/docker/volumes/coconikki_storage/_data`
- sudo権限でのrclone実行（環境変数の引き継ぎ）

### Phase 2: 復元スクリプト作成

バックアップを取るだけでなく、復元できることが重要です。

**やったこと**:
- Google Driveからのダウンロード
- データベースのDROP & CREATE
- 既存ストレージの退避（念のため）
- 復元後のアプリケーション再起動
- 安全確認プロンプト（誤実行防止）

**学び**:
- 復元は破壊的操作なので、必ず確認プロンプトを入れる
- 最新バックアップだけでなく、特定日時のバックアップも選べるようにする
- 既存データの一時退避により、復元失敗時のリカバリが可能

**ハマったポイント**:
- PostgreSQLの`DROP DATABASE`は接続中だと失敗する
- Kamalでのアプリ再起動コマンド（`bin/kamal app restart`）

### Phase 3: Discord通知の実装

バックアップが成功したか失敗したかを知る必要があります。

**やったこと**:
- Discord Webhookの作成
- リッチembedでの通知（色分け、詳細情報）
- 成功時: 緑色、実行日時、サイズ情報、Google Driveリンク
- 失敗時: 赤色、エラー内容

**学び**:
- Discord WebhookはシンプルなJSONで簡単に実装できる
- embedのcolorは数値（緑=3066993、赤=15158332）
- curlで`--silent --show-error`を使うとログがきれい

**ハマったポイント**:
- JSONのエスケープ処理（HEREDOCで解決）
- 動的なフィールド追加（エラー時のみ表示）

### Phase 4: 自動実行設定（cron）

**やったこと**:
- cronで毎日午前3時に自動実行
- 環境変数の引き継ぎ（`sudo -E`）
- ログファイルへの出力（`~/backup.log`）

**学び**:
- cronは環境変数が限定的なので、必要な変数を明示的にexport
- `sudo -E`で環境変数を引き継げる
- 午前3時はアクセスが少なく、バックアップに適した時間帯

**ハマったポイント**:
- cron内でのDISCORD_WEBHOOK_URLの設定
- タイムゾーンの確認（UTC vs JST）

### Phase 5: ドキュメント整備

**やったこと**:
- 運用マニュアル（operations.md）
- 復元ガイド（restore-guide.md）
- セットアップ手順（setup.md）
- ドキュメント目次（README.md）

**学び**:
- ドキュメントは「使う人の視点」で分ける
  - 日常運用する人 → operations.md
  - 復元が必要な人 → restore-guide.md
  - 新規構築する人 → setup.md
- トラブルシューティングは実際にハマったポイントを記載
- スクリーンショットではなく、コマンド例を豊富に

## 🎓 学んだこと

### 技術的な学び

1. **rcloneの便利さ**
   - 多数のクラウドストレージに統一的にアクセス可能
   - OAuth認証もコマンドラインで完結
   - `--min-age`で古いファイル削除が簡単

2. **Bashスクリプトのベストプラクティス**
   - `set -e`でエラー時に即座に停止
   - trapでクリーンアップ処理を確実に実行
   - HEREDOCで複雑な文字列を扱いやすく

3. **PostgreSQLバックアップ**
   - `pg_dump`はシンプルで信頼性が高い
   - gzip圧縮でサイズを大幅削減（数MBレベル）

4. **Docker環境でのバックアップ**
   - Dockerボリュームへのアクセス方法
   - ホストとコンテナ間のデータ受け渡し

### 設計上の学び

1. **マルチバックエンド設計の価値**
   - 最初からgdrive/local/s3に対応
   - 将来的な移行が容易
   - テストもしやすい

2. **通知の重要性**
   - バックアップは「取れている」ことを確認できなければ意味がない
   - Discord通知により、日々の安心感が得られる

3. **ドキュメントファースト**
   - 実装と並行してドキュメントを書くことで、設計の穴に気づく
   - 将来の自分への投資

### 運用上の学び

1. **自動化の価値**
   - 手動バックアップは絶対に忘れる
   - 一度設定すれば、あとは確認するだけ

2. **復元テストの重要性**
   - バックアップが取れていても、復元できなければ意味がない
   - 定期的な復元テストを開発環境で実施すべき

3. **段階的な導入**
   - Phase 1: 基本機能
   - Phase 2: 復元機能
   - Phase 3: 通知
   - Phase 4: 自動化
   - Phase 5: ドキュメント

   各Phaseで動作確認しながら進めることで、問題の切り分けが容易

## 🚀 今後の展望

### 短期的改善案

1. **復元テストの自動化**
   - 月1回、開発環境で自動復元テスト
   - テスト結果をDiscord通知

2. **バックアップの増分化**
   - 現在は全量バックアップ
   - データ量が増えたら増分バックアップも検討

3. **複数世代のバックアップ**
   - 現在は7日分
   - 月初のバックアップを長期保存（例: 3ヶ月分）

### 長期的展望

1. **オフサイトバックアップの多重化**
   - Google Drive + AWS S3のような多重バックアップ
   - 異なるプロバイダーでリスク分散

2. **バックアップ暗号化**
   - 現在は非暗号化
   - rclone cryptでの暗号化バックアップ

3. **Point-in-Time Recovery (PITR)**
   - PostgreSQLのWALログを使った秒単位の復元
   - より細かい復元ポイント
