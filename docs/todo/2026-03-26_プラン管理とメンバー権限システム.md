# プラン管理とメンバー権限システム

**作成日**: 2026年3月26日
**ステータス**: 設計完了、実装待ち

---

## 概要

coconikkiに「ユーザープラン」と「メンバー権限」の2軸の権限管理システムを導入する。

### 背景
- 現状、交換日記のメンバー数に制限がない
- 全ユーザーが同じ制限（スレッド数、ストレージ等）
- メンバー全員が同じ権限（role: "writer"のみ）
- 将来的な有料プラン導入を見据えた設計が必要

---

## 決定事項

### 1. ユーザープラン（User model）

#### プランの種類
```ruby
enum :plan, {
  free: "free",      # 無料ユーザー（デフォルト）
  paid: "paid",      # 有料ユーザー
  tester: "tester"   # テスター（有料相当の権限、内部識別用）
}
```

#### プラン別制限

**無料プラン**
```ruby
MAX_THREADS_FREE = 10                    # 参加可能スレッド数
MAX_STORAGE_FREE = 100.megabytes         # ストレージ容量
MAX_MEMBERS_PER_THREAD_FREE = 4          # 作成したスレッドのメンバー上限
MAX_POSTS_PER_HOUR = 5                   # 1時間あたりの投稿数（変更なし）
MAX_POSTS_PER_DAY = 20                   # 1日あたりの投稿数（変更なし）
```

**有料・テスタープラン**
```ruby
MAX_THREADS_PAID = 50                    # 参加可能スレッド数
MAX_STORAGE_PAID = 1.gigabyte            # ストレージ容量
MAX_MEMBERS_PER_THREAD_PAID = 10         # 作成したスレッドのメンバー上限
# 投稿頻度制限は無料と同じ（スパム対策として維持）
```

### 2. メンバー権限（Membership model）

#### 権限の種類
```ruby
enum :role, {
  writer: "writer",  # 参加者（投稿のみ）
  admin: "admin"     # 管理者（メンバー管理、設定変更）
}
```

#### 権限別の機能

**参加者（writer）**
- 投稿作成・編集・削除
- スレッド閲覧
- 自分の脱退

**管理者（admin）**
- 参加者の全機能
- メンバー削除
- 他メンバーを管理者に昇格
- 順番変更（新規実装）
- スレッド設定編集
- スレッド削除

**特記事項**
- スレッド作成者は自動的に`admin`に設定
- 将来的に`owner`権限を追加予定（Phase 2）

## スレッドのメンバー上限ルール

### 採用案: **案1（作成者のプランで制限）**

```ruby
# CorrespondenceThread model
def max_members
  creator = memberships.find_by(position: 1)&.user
  return User::MAX_MEMBERS_PER_THREAD_FREE unless creator

  creator.paid_or_tester? ? User::MAX_MEMBERS_PER_THREAD_PAID : User::MAX_MEMBERS_PER_THREAD_FREE
end
```

### ルールの詳細
- スレッド作成者が無料ユーザー → 4人まで
- スレッド作成者が有料/テスター → 10人まで
- 作成者のプランに依存（シンプルでわかりやすい）

### メリット
- 実装がシンプル
- ユーザーにわかりやすい訴求（「有料にすると新しく作るスレッドは10人まで可能！」）
- 将来の拡張パス明確（スレッド自体にプランを持たせる案3へ移行可能）

### 検討した他の案
- **案2**: 管理者の中に1人でも有料ユーザーがいれば上限緩和
  - メリット: 公平
  - デメリット: 複雑、ダウングレード時の処理が必要
- **案3**: スレッド自体にプランを持たせる
  - メリット: 最も柔軟
  - デメリット: 実装が重い、Phase 2以降で検討
